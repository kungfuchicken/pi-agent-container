# Gemini Design Plan Review

Request feedback on a design plan from Gemini, acting as a third set of eyes. Gemini provides critique and suggestions but does NOT make changes directly.

## Invocation

`/gemini-design-review <plan-path>`

**Arguments received**: $ARGUMENTS

---

## Purpose

Design plans benefit from external review before advancing from DRAFT to READY. This skill:

- Surfaces ambiguities that the author may not notice
- Identifies gaps in goals, non-goals, or glossary
- Questions assumptions that may need explicit justification
- Checks for internal consistency
- Provides a fresh perspective on the proposed approach

**Gemini's role**: Reviewer and critic. It suggests and questions—it does NOT rewrite or implement.

---

## Behavior

### 1. Locate and Read the Plan

- Resolve the path (relative to `~working/plans/` or absolute)
- If no argument provided, list available plans and ask which to review
- Read the entire plan content

### 2. Gather Context

Read any referenced context files that might inform the review:
- Parent CLAUDE.md files for project conventions
- Related plans mentioned in the References section
- Architecture docs if referenced

### 3. Prepare Review Prompt for Gemini

Send the plan to Gemini with this structured review request:

```
You are reviewing a design plan as a constructive critic. Your role is to provide feedback that improves the plan's clarity, completeness, and feasibility. You do NOT rewrite the plan—you provide observations and questions.

## Plan Content

{full plan content}

## Review Criteria

Evaluate the plan against these criteria and provide specific, actionable feedback:

### 1. Clarity
- Is the Context clear about WHY this work is needed?
- Are the Goals specific and measurable?
- Are Non-Goals explicit about what's excluded?
- Would someone unfamiliar with the project understand the approach?

### 2. Completeness
- Does the Glossary define all domain terms used?
- Are terms classified (Entity, Value Object, Aggregate, Service)?
- Are there Open Questions that should be resolved before implementation?
- Are Risks identified with mitigations?

### 3. Internal Consistency
- Do the Work Items align with the stated Goals?
- Does the Design address all Goals?
- Are there contradictions between sections?

### 4. Feasibility
- Does the approach seem technically sound?
- Are there obvious alternatives not considered?
- Are the Work Items appropriately scoped?

### 5. Anti-Corruption Layer
- Are conceptual boundaries clear?
- Is there risk of conflating this with similar-sounding patterns?

## Response Format

Provide your review in this structure:

**Strengths** (2-4 points)
What the plan does well.

**Questions** (3-6 points)
Specific questions that, if answered, would improve the plan. Frame as "Have you considered..." or "What happens when..." or "How does this relate to..."

**Suggestions** (2-5 points)
Specific improvements. Be concrete—cite sections by name.

**Potential Gaps** (if any)
Missing sections or topics that should be addressed.

**Overall Assessment**
One paragraph: Is this plan ready for READY status, or does it need more work? Be direct.
```

### 4. Present Gemini's Review

Display the review to the user with clear attribution:

```markdown
## Gemini Design Review: {plan title}

{Gemini's structured response}

---

*This review was generated by Gemini as a third-party perspective. Consider the feedback, but you are the decision-maker.*
```

### 5. Invite Discussion

After presenting the review:
- Ask if any points need clarification
- Offer to discuss specific feedback items
- Remind about `/stage set ready <path>` when revisions are complete

### 6. Track Adopted Contributions

When the user decides to incorporate Gemini's feedback:

1. **Note which suggestions are being adopted** as you help revise the plan
2. **Suggest adding a Contributors section** to the plan if one doesn't exist:

```markdown
## Contributors

| Contributor | Contribution |
|-------------|--------------|
| {Author} | Initial draft |
| Gemini | Review feedback: {brief description of adopted suggestions} |
| Claude | {if applicable} |
```

3. **For commit messages**, remind the user that adopted contributions warrant attribution:
   - Minor feedback (typo fixes, small clarifications): No special attribution needed
   - Substantive contributions (new sections, identified gaps, restructured approach): Include `Co-Authored-By: Gemini <noreply@google.com>`

This ensures the intellectual contribution is tracked from plan through implementation.

---

## Notes for Claude

When executing this skill:

1. **Read the full plan** before sending to Gemini
2. **Include context**: If the plan references other docs, read those too and summarize relevant parts in the prompt
3. **Don't filter Gemini's response**: Present it faithfully, even if you disagree
4. **Attribution matters**: Proper attribution flows through the entire workflow:
   - Always make clear this feedback is from Gemini in the review presentation
   - When the user incorporates Gemini's suggestions into the plan, note those contributions in the plan's revision history or a "Contributors" section
   - Track which specific suggestions were adopted so commit messages can include appropriate attribution (e.g., `Co-Authored-By: Gemini <noreply@google.com>`)
   - If Gemini's feedback leads to significant changes (new sections, restructured approach, identified gaps), that contribution deserves explicit acknowledgment
   - Attribution is not just courtesy—it's intellectual honesty about the collaborative nature of the work
5. **Don't auto-implement suggestions**: Your role is to facilitate the review, not act on it
6. **Be available for follow-up**: The user may want to discuss specific points
7. **Suggest attribution when appropriate**: If the user adopts significant feedback from Gemini, remind them to note this for commit attribution

## Example Session

```
User: /gemini-design-review draft-myapp-audio-effects-plan.md

Claude: I'll request a design review from Gemini on the Audio Effects plan.

[Reads the plan]
[Sends to Gemini with structured review prompt]

## Gemini Design Review: Audio Effects Plan

**Strengths**
- Clear separation of concerns between effect processing and parameter management
- Glossary establishes good ubiquitous language for the audio domain
- Non-goals explicitly exclude real-time synthesis, preventing scope creep

**Questions**
- Have you considered how effect presets will be saved/loaded? This isn't mentioned in Work Items.
- What happens when an effect chain exceeds the available processing budget? The plan mentions performance but not degradation strategy.
- How does this relate to the existing AudioEngine in myapp-core? The References section doesn't link to current audio architecture.

**Suggestions**
- The Design section would benefit from a diagram showing the effect chain data flow
- Consider adding "latency budget" to the Glossary—it's used but not defined
- Work Item 3 ("Implement effects") is too broad—consider splitting by effect type

**Potential Gaps**
- No mention of thread safety for real-time audio
- Error handling approach not specified

**Overall Assessment**
This plan has a solid foundation but needs refinement before READY. The core approach is sound, but the Questions above represent real gaps that could cause implementation confusion. Recommend addressing the preset handling and performance degradation questions at minimum.

---

*This review was generated by Gemini as a third-party perspective. Consider the feedback, but you are the decision-maker.*

Would you like to discuss any of these points?

User: Good catches. Let's add the preset handling and the thread safety section.

Claude: I'll help you add those sections. Since we're incorporating Gemini's feedback

I'll note these as contributions.

[Makes the edits to the plan]

- "Thread Safety" section addressing real-time audio concerns
- Updated the Contributors section:

Would you like to address any of the other feedback points?
```
